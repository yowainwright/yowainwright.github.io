# Anti-Slop Check

Detect and remove common AI-generated bloat: unnecessary emojis, useless comments, redundant docs, and non-standard patterns.

## Usage

```
/anti-slop src/
/anti-slop --fix src/utils.ts
/anti-slop --check  # dry run, report only
```

## What is Slop?

AI-generated code often includes:
- Emojis in code/comments/commits
- Obvious/redundant comments
- Over-documented simple code
- Unnecessary README files
- Boilerplate config nobody asked for
- Tests that don't test anything meaningful

## Detection Rules

### 1. Emoji Pollution

```typescript
// SLOP - emojis in code
console.log("✅ Success!")
// 🚀 Initialize the server
const server = createServer()

// CLEAN
console.log("Success")
// Initialize the server
const server = createServer()
```

**Grep patterns:**
```bash
# Find emoji usage in code
rg '[\x{1F300}-\x{1F9FF}]' --type ts
rg '[\x{2600}-\x{26FF}]' --type ts
```

**Allowed exceptions:**
- User-facing UI strings (if explicitly requested)
- Test fixtures for emoji handling
- i18n/l10n content

### 2. Useless Comments

#### Obvious Comments
```typescript
// SLOP
// Initialize the counter
let counter = 0

// Loop through the array
for (const item of items) { }

// Return the result
return result

// CLEAN - no comment needed, code is self-evident
let counter = 0
for (const item of items) { }
return result
```

#### Redundant JSDoc
```typescript
// SLOP - adds nothing
/**
 * Gets the user
 * @param id - The user id
 * @returns The user
 */
function getUser(id: string): User

// CLEAN - only document non-obvious behavior
function getUser(id: string): User

// ACCEPTABLE - documents edge case
/**
 * Returns null if user is soft-deleted
 */
function getUser(id: string): User | null
```

#### Section Dividers
```typescript
// SLOP
// ============================================
// UTILITY FUNCTIONS
// ============================================

// ----------------------------------------
// Helper Methods
// ----------------------------------------

/* ****************************
 * API HANDLERS
 * ****************************/

// CLEAN - use file organization instead
// utils.ts, handlers.ts, etc.
```

#### AI Attribution Comments
```typescript
// SLOP
// Generated by AI
// Created with assistance from Claude
// AI-generated code below

// CLEAN - remove, use git history for attribution
```

### 3. Useless Markdown Files

#### Unnecessary READMEs
```
# SLOP - empty/useless READMEs
src/
├── components/
│   └── README.md  # "This folder contains components"
├── utils/
│   └── README.md  # "Utility functions"
└── README.md      # Just repeats folder structure

# CLEAN - only root README with actual content
README.md  # Installation, usage, contributing
```

#### Over-documented Internal Code
```markdown
# SLOP - docs/api/internal-utils.md
## Internal Utilities
These are internal utilities used by the codebase...

# CLEAN - code should be self-documenting
# Only document public API and architecture decisions
```

### 4. Useless Tests

#### Tautological Tests
```typescript
// SLOP - tests nothing meaningful
test("returns true when condition is true", () => {
  expect(true).toBe(true)
})

test("adds numbers", () => {
  expect(1 + 1).toBe(2)
})

test("array has length", () => {
  const arr = [1, 2, 3]
  expect(arr.length).toBe(3)
})
```

#### Implementation Tests
```typescript
// SLOP - tests implementation, not behavior
test("calls the database", () => {
  getUser("123")
  expect(db.query).toHaveBeenCalled()
})

// CLEAN - tests actual behavior
test("returns user when found", async () => {
  const user = await getUser("123")
  expect(user.id).toBe("123")
})
```

#### Snapshot Everything
```typescript
// SLOP - snapshots for everything
test("renders", () => {
  expect(render(<Button />)).toMatchSnapshot()
})

// CLEAN - snapshots only for complex, stable output
test("renders email template correctly", () => {
  expect(renderEmail(data)).toMatchSnapshot()
})
```

### 5. Non-Standard Config

#### Unnecessary Config Files
```
# SLOP - config files nobody asked for
.prettierrc
.prettierignore
.eslintrc.js
.eslintignore
.stylelintrc
.huskyrc
.lintstagedrc
commitlint.config.js

# CLEAN - minimal, project-appropriate config
oxlint.json  # fastest (recommended)
biome.json   # OR oxlint (not both, unless biome for formatting only)
tsconfig.json
```

**Linter preference:** oxlint > biome > eslint (speed order)

#### Over-Configured Defaults
```json
// SLOP - biome.json with all defaults spelled out
{
  "formatter": {
    "enabled": true,
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 80
  },
  "linter": {
    "enabled": true,
    "rules": {
      "recommended": true
    }
  }
}

// CLEAN - only non-defaults
{
  "formatter": {
    "lineWidth": 100
  }
}
```

### 6. Verbose Error Messages

```typescript
// SLOP
throw new Error("An error occurred while attempting to process the user data. Please try again later or contact support if the issue persists.")

// CLEAN
throw new Error("Failed to process user data")
```

### 7. Over-Logging

```typescript
// SLOP
console.log("Starting function...")
console.log("Processing item:", item)
console.log("Item processed successfully")
console.log("Function complete!")

// CLEAN - log meaningful events only
logger.info("Processed batch", { count: items.length })
```

## Grep Patterns

```bash
# Emojis in code
rg '[\x{1F300}-\x{1F9FF}]|[\x{2600}-\x{26FF}]|[\x{2700}-\x{27BF}]' --type ts

# Obvious comments
rg '// (Loop|Iterate|Return|Initialize|Set|Get|Create|Check|Verify)' --type ts

# Section dividers
rg '^[ \t]*/[/*][ \t]*[-=*]{3,}' --type ts

# AI attribution
rg -i '(generated|created|assisted).*(ai|claude|gpt|copilot)'

# Empty/short READMEs
find . -name "README.md" -size -200c

# Console.log spam
rg 'console\.(log|info|debug)\(' --type ts -c | sort -t: -k2 -nr
```

## Checklist

- [ ] No emojis in code, comments, or commits
- [ ] No obvious/redundant comments
- [ ] No JSDoc on self-evident functions
- [ ] No section divider comments
- [ ] No AI attribution comments
- [ ] No useless README files in subdirectories
- [ ] No over-documentation of internal code
- [ ] No tautological tests
- [ ] No duplicate/conflicting config files
- [ ] No verbose error messages
- [ ] No excessive logging

## Philosophy

**Good code is self-documenting.** If you need a comment to explain what code does, refactor the code instead. Comments should explain *why*, not *what*.

**Less is more.** Every file, comment, and line of config is maintenance burden. Only add what provides clear value.

**Follow the project's patterns.** Don't introduce new tools, configs, or conventions without explicit approval.
